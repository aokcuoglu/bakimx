# BakımX v0.5.7 — Manual QA Checklist

Run against a deploy with a real database and `SESSION_SECRET` + `CRON_SECRET` set.
Mark each item ✅/❌. Items map to the v0.5.7 security fixes.

## Pre-req
- [ ] App is reachable behind nginx; nginx sets `X-Forwarded-For` to the real client IP (see README).
- [ ] `CRON_SECRET` and `SESSION_SECRET` (≥32 chars) configured in the environment.

## 1. Public registration removed
- [ ] Visiting `/register` redirects to `/login` (no registration form shown).
- [ ] `curl -i -X POST <host>/api/auth/register -F email=x@y.z -F password=secret123` returns **404** and creates no account.
- [ ] No "Kayıt Ol / Sign up" link is reachable from the login screen.

## 2. Login works
- [ ] Login with demo credentials (`demo@bakimx.com`) succeeds and lands on `/app`.
- [ ] Logout returns to `/login`; visiting `/app` while logged out redirects to `/login`.

## 3. Login rate limit
- [ ] Submit wrong password ≥9 times within a minute from one client → eventually **429** with a "Çok fazla deneme" message and `Retry-After` header.
- [ ] After the window resets (~60s), login works again.

## 4. Account enumeration / safe errors
- [ ] Login with an **unknown** e-mail and with a **known e-mail + wrong password** both return the **same** message: `E-posta adresi veya şifre hatalı`.
- [ ] Response timing for unknown-email vs wrong-password is similar (no fast "not found" path).

## 5. Cron auth
- [ ] `GET /api/cron/reminders` with **no** Authorization header → **401**.
- [ ] With a **wrong** Bearer token → **401**.
- [ ] With the correct `Authorization: Bearer <CRON_SECRET>` → **200** and reminders process.
- [ ] (Config) With `CRON_SECRET` unset on the server → endpoint returns **500** (never open).

## 6. Status transition guards (no OTP/approval bypass)
- [ ] `POST /api/intakes/<id>/status` with `{"status":"approved"}` is **rejected** ("Onay yalnızca müşteri doğrulaması (OTP) ile verilebilir").
- [ ] `POST /api/intakes/<id>/status` with a garbage value (e.g. `"foo"`) is **rejected** ("Geçersiz durum").
- [ ] Normal intake flow still works: draft → waiting_approval → (OTP verify) approved → in_progress → ready_for_delivery → delivered.
- [ ] Order status buttons in the UI still work for all offered transitions; an illegal jump (e.g. delivered → draft) is rejected.
- [ ] `POST /api/orders/<id>/payment` with an invalid paymentStatus is rejected.

## 7. Public intake share `/s/[token]` safe
- [ ] Page renders for a valid token; an invalid/expired/inactive token shows not-found.
- [ ] No internal data leaks: internal notes, OCR raw text, internal IDs, OTP codes are absent.
- [ ] Internal timeline events (internal note added, labor session start/stop) do **not** appear in the timeline.

## 8. Public passport `/p/[token]` safe
- [ ] Page renders for a valid token; invalid/expired token → not-found.
- [ ] No internal data leaks (as above).

## 9. Print/PDF routes escaped (XSS)
- [ ] Create a customer/vehicle/intake with an XSS payload in a free-text field (e.g. customer name or complaint = `<img src=x onerror=alert(1)>`), generate a share link, open `/s/<token>/pdf` and `/p/<token>/pdf`.
- [ ] The payload renders as **literal text** (escaped); no script executes, no broken markup.
- [ ] Same check for a damage note, an order item name, and the workshop name / custom PDF template.

## 10. Hidden photo / payment flags respected
- [ ] Share with `showPhotos = off`: no photos render on `/s/[token]`, and `GET /s/<token>/photos/<photoId>` returns **not-found**.
- [ ] Share with `showOrderItems = on, showPaymentStatus = off`: item names/quantities are visible but **no prices, no line totals, no grand total** on `/s/[token]` and `/s/[token]/pdf`; the WhatsApp share text contains no total.
- [ ] Same share with `showPaymentStatus = on`: prices and totals reappear.
- [ ] Passport with `showWorkOrders = on, showPaymentStatus = off`: work orders listed without prices/grand total.

## 11. App-shell regression
- [ ] Collapse the sidebar, reload the page → it stays collapsed; expand, reload → stays expanded.
- [ ] No React hydration mismatch warning in the browser console on `/app`.

## Sign-off
- [ ] All blocking items pass → approve for release.
