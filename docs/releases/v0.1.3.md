# BakimX v0.1.3 — Image Storage & Intake Media Foundation

**Release Date:** June 2026
**Previous Release:** v0.1.2 — Public Output & PDF Foundation

---

## Summary

v0.1.3 implements persistent intake media storage foundation using a clean storage provider abstraction. Vehicle intake photos now actually persist through server-side storage flows, with tenant-scoped paths, file validation, signed URL display, and full audit logging. The app works out-of-the-box with the mock provider (no storage env vars needed) and can be configured for Supabase Storage when ready.

---

## Changes

### 1. Storage Provider Architecture (`lib/storage/`)

- **`types.ts`** — Extended with:
  - `ALLOWED_MIME_TYPES`: `image/jpeg`, `image/png`, `image/webp`
  - `MAX_FILE_SIZE_BYTES`: 8 MB
  - `STORAGE_PATH_PREFIX`: `workshops`
  - `validateUploadFile()`: MIME type and size validation with Turkish error messages
  - `buildStoragePath()`: Tenant-scoped path builder with safe filename normalization
  - HEIC files rejected with clear message
- **`mock-storage-provider.ts`** — Base64 in-memory storage (unchanged, safe for local dev)
- **`supabase-storage-provider.ts`** — Full implementation using `@supabase/supabase-js`:
  - Upload, delete, signed URL
  - Takes pre-created client (no `require()` in module)
  - Bucket configurable via `SUPABASE_STORAGE_BUCKET`
- **`s3-storage-provider.ts`** — Typed placeholder, throws "not yet implemented" when selected
- **`storage-provider.ts`** — Async factory using `STORAGE_PROVIDER` env var:
  - `mock` (default) → MockStorageProvider
  - `supabase` → dynamic import + SupabaseStorageProvider
  - `s3` → dynamic import + S3StorageProvider (error)
  - Singleton pattern with `resetStorageProvider()` for testing
- **`index.ts`** — Barrel export for all types, functions, constants

### 2. Supabase Storage Integration

- `@supabase/supabase-js` added as dependency
- Upload flow: server action receives `File` → validates → uploads via provider → stores metadata in DB
- Service role key never exposed to browser (all uploads server-side)
- Signed URLs generated on-demand for display (never stored permanently)
- Storage path format: `workshops/{workshopId}/intakes/{intakeFormId}/{photoType}/{photoId}-{safeFileName}`
- Bucket default: `bakimx-media` (configurable via `SUPABASE_STORAGE_BUCKET`)

### 3. S3-Compatible Provider Preparation

- `S3StorageProvider` class implements `StorageProvider` interface
- All methods throw: "S3 depolama sağlayıcısı henüz uygulanmadı"
- Environment variables documented in `.env.example`
- Does not break mock or Supabase providers

### 4. Vehicle Photo Upload Persistence

- **Prisma schema updated:** `VehiclePhoto` model now has:
  - `fileName: String?`
  - `mimeType: String?`
  - `sizeBytes: Int?`
  - `storageProvider: String?`
  - `storageKey: String?`
  - `updatedAt: DateTime @updatedAt`
- Backward compatible: all new fields nullable, existing records with null fileUrl still work
- `addPhotoAction` now:
  - Validates file MIME type and size
  - Uploads file through storage provider
  - Persists all metadata
  - Creates audit log entry
- Photo capture UI now sends actual `File` via FormData (was only text fields before)
- File input `accept` updated: `image/jpeg,image/png,image/webp` (not `image/*`)

### 5. Photo Replace and Delete

- **New action:** `replacePhotoAction(formData)` — PUT
  - Validates new file
  - Deletes old storage object
  - Uploads new file
  - Updates all metadata fields
  - Audit log: `photo_replaced`
- **Updated action:** `removePhotoAction(photoId, intakeFormId)` — DELETE
  - Deletes storage object if storageKey exists
  - Deletes database record
  - Audit log: `photo_deleted`
- Error handling: `photo_upload_error`, `photo_replace_error` audit entries

### 6. Authenticated Photo Viewing

- **New API route:** `GET /api/photos?id=`
  - Requires auth, validates workshopId
  - Returns blob (mock) or redirects to signed URL (Supabase)
  - Cache-Control: private, max-age=300
- Protected in middleware: `/api/photos` added to protected API paths

### 7. Public-Safe Photo Rendering

- **New route:** `GET /s/[token]/photos/[photoId]`
  - Token-based access (no auth required)
  - Validates share link is active and not expired
  - Validates photo belongs to share link's intake/workshop
  - Returns blob (mock) or signed URL redirect (Supabase)
  - No storage keys, internal IDs, or notes exposed
- **Public share page updates:**
  - Photo thumbnails rendered in 3-column grid for photos with files
  - Loading spinner while photo loads
  - Professional placeholder (ImageOff icon) for missing/failed photos
  - Photo checklist text still shows all photo types with status
  - Removed "Fotoğraf önizlemesi depolama entegrasyonu sonrası gösterilecektir." (photos now render!)

### 8. Authenticated Intake Media Gallery

- `IntakeDetail` photos tab redesigned:
  - **Photo checklist** (required/optional) with capture, preview, save, replace, remove
  - **"Fotoğraf Ekle" card** with file type selector, camera/file picker, note, format info
  - **"Kaydedilmiş Fotoğraflar" gallery** — grid cards with:
    - Photo thumbnail (loaded via `/api/photos`)
    - Type label
    - File name, size (formatted), MIME type
    - "Tekrar çek / değiştir" button
    - Delete button
    - Loading overlay during replace
  - Upload button shows loader icon during upload
  - All buttons touch-friendly (min 44px target)
- `PhotoThumbnail` component: fetches photo via API, shows loading/failure states
- `PhotoGalleryCard` component: card layout with metadata and actions

### 9. Image Validation and Compression Strategy

- MIME type validation: only JPEG, PNG, WebP allowed
- HEIC explicitly rejected with Turkish error message
- File size validation: max 8 MB
- Client-side compression not yet implemented (documented as future scope)
- `accept` attribute on file inputs: `image/jpeg,image/png,image/webp`

### 10. Tenant Isolation and Security

- Every upload verifies `workshopId` from authenticated user
- Every media mutation verifies `workshopId`
- Storage paths always include `workshopId` and `intakeFormId`
- Client-provided `workshopId` never trusted
- Service role credentials never exposed to browser
- Public photo route validates token → intake → photo ownership
- No raw storage keys in API responses to unauthenticated users
- Photo IDs only accessible through token validation on public routes

### 11. Audit Logs

New audit actions:
- `photo_uploaded` — with type, storageProvider, sizeBytes
- `photo_replaced` — with type, storageProvider, sizeBytes
- `photo_deleted` — with type, hadStorageKey
- `photo_upload_error` — with error message
- `photo_replace_error` — with error message

### 12. Environment Variables

New in `.env.example`:
- `STORAGE_PROVIDER=mock`
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_STORAGE_BUCKET` (commented, for Supabase)
- `S3_ENDPOINT`, `S3_REGION`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`, `S3_BUCKET`, `S3_FORCE_PATH_STYLE` (commented, placeholder)

## Known Limitations

- Real SMS provider not integrated (mock/demo mode)
- Real OCR/VIN extraction not implemented
- WhatsApp Business API not integrated (manual share link only)
- S3 storage provider not implemented (typed placeholder)
- HEIC image format not supported (rejected with error message)
- Client-side image compression not implemented (file size limit enforced, future scope)
- @react-pdf/renderer not yet generating server-side PDF (print-optimized HTML available)
- No payment/billing module
- No Docker files (by design)
