# BakimX v0.3.1 — OCR & Smart Capture Foundation

**Date:** 2026-06-12

## Summary

BakimX v0.3.1 introduces the OCR & Smart Capture foundation for workshops. Users can upload or photograph a Turkish vehicle registration certificate (ruhsat), have OCR extract owner and vehicle fields, review/edit the extracted data, then confirm to auto-create or update customer and vehicle records. The flow integrates with the intake and work order creation pipeline.

## Features

### OCR Provider Architecture

- `OcrProvider` interface with `RegistrationOcrResult` type and per-field confidence scores
- **Mock provider** (`OCR_PROVIDER=mock`) — returns hardcoded Turkish registration data with 1.5s delay simulation
- **DeepSeek provider** (`OCR_PROVIDER=deepseek`, default) — Tesseract.js for text extraction + DeepSeek chat API for field parsing with retry logic
- **OpenAI provider** (`OCR_PROVIDER=openai`) — OpenAI Responses API with structured JSON schema output and vision input
- Async factory `getOcrProvider()` selects provider via `OCR_PROVIDER` env var
- Graceful fallback: when `OCR_PROVIDER=mock` or when no API key is set, provides clear Turkish error messages

### Smart Capture Registration Flow (`/app/smart-capture/registration`)

- **Mobile-first camera/file upload**
  - `capture="environment"` for direct camera capture on mobile
  - Drag & drop support
  - File upload button
  - MIME type validation (JPEG, PNG, WebP, HEIC/HEIF) and 8 MB size limit
  - Server-side HEIC-to-JPEG conversion via `heic-convert`
- **Image preview** — uploaded registration photo shown after OCR processing
- **Processing state** — animated scan indicator during OCR
- **Error state** — Turkish error messages for all failure modes

### Extracted Fields

OCR result structure with per-field confidence:
- `plate` — License plate
- `vin` — Chassis/VIN number
- `ownerName` / `ownerSurname` — Owner name/surname
- `brand` / `model` — Brand/model
- `vehicleType` — Vehicle type (e.g., Binek, Kamyonet)
- `modelYear` — Model year
- `engineNo` — Engine number
- `registrationDate` — Registration date
- `rawText` — Full OCR text (stored in DB, not returned to client)

### User Confirmation & Editing

- Extracted fields shown in editable form with warning banner
- All fields editable before confirmation
- Phone number required (min 10 digits) for customer matching
- "Onayla ve Kaydet" (Confirm & Save) creates/updates records
- "Yeniden Oku" (Re-read) returns to upload step

### Record Creation / Prefill (confirm route)

- Finds existing vehicle by plate or VIN (with duplicate detection)
- If matching vehicle found: updates empty fields, optionally reassigns customer
- If no matching vehicle: finds or creates customer by phone, creates new vehicle
- `vinConfirmed` flag set when VIN is 17 characters
- Creates `OcrLog` audit entry linking user, customer, vehicle
- Returns customer/vehicle IDs and direct intake URL with `source=registration` param

### Work Order Integration

- `/app/orders/new` — "Ruhsattan Doldur" card linking to smart capture
- `/app/vehicles/new` — "Ruhsattan Oku" link (now functional, not placeholder)
- `/app/intakes/new?customerId=...&vehicleId=...&source=registration` — skips customer/vehicle steps in intake wizard
- Sidebar navigation: "Ruhsat Okuma" entry

### OCR Audit Log (OcrLog model)

- `ocrProvider` — which provider was used
- `rawText` — raw OCR text (not returned in API responses)
- `extractedJson` / `confirmedJson` — before/after field values
- `confirmedAt` — when user confirmed
- `userId` / `workshopId` — who and which workshop
- `customerId` / `vehicleId` / `serviceOrderId` — linked records

### Security / KVKK

- OCR data scoped to workshop (`workshopId`)
- Raw OCR text not returned in API responses, stored only in DB
- User confirmation required before any record creation
- Turkish information notices throughout the flow

## Known Limitations

1. OCR_PROVIDER defaults to `mock` — returns fixed demo data, not real extraction. Real OCR requires `OCR_PROVIDER=deepseek` or `OCR_PROVIDER=openai` with corresponding API key and model env vars
2. Both `DEEPSEEK_OCR_MODEL` and `OPENAI_OCR_MODEL` are required when using their respective providers (no hardcoded defaults)
3. Tesseract.js worker is a lazy singleton; `terminateTesseractWorker()` is exported for graceful shutdown but not called automatically
4. Large images sent as base64 data URLs in JSON body (no multipart upload yet); server-side content-length check caps at 12 MB request body
5. `vinOcrRaw` field on Vehicle model is defined but not populated in v0.3.1
6. OpenAI provider uses Responses API (`/v1/responses`) — requires compatible model
7. HEIC conversion adds processing overhead; preview data URL stored in memory
8. Vehicle plate is unique per workshop (`@@unique([workshopId, plate])`); VIN is indexed but not unique per workshop (same VIN can exist across workshops)
9. Currency is TRY only
10. No real SMS/WhatsApp/payment/invoice integration (all mocks or client-side links)

## Database Migration Required

This patch adds a unique constraint on `Vehicle(workshopId, plate)` and an index on `Vehicle(workshopId, vin)`. If you have duplicate plates within the same workshop, you must resolve them before running the migration.

```bash
# Check for duplicate plates before migrating:
# SELECT "workshopId", plate, COUNT(*) FROM "Vehicle" GROUP BY "workshopId", plate HAVING COUNT(*) > 1;

# Option 1: Prisma migrate (recommended)
npx prisma migrate dev --name add_vehicle_plate_unique_constraint

# Option 2: If setting up fresh
npx prisma migrate deploy

# Option 3: Development only
npx prisma db push
```
