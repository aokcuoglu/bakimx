# BakimX v0.3.3 — Smart Capture Hardening & Real OCR Provider

**Date:** 2026-06-12

## Summary

BakimX v0.3.3 hardens the OCR & Smart Capture flow with explicit real OCR provider selection, improved image upload validation, confidence-based UX, rawText privacy, VIN/plate duplicate warnings, and Tesseract-only provider support. Mock OCR remains the safe default.

## Features

### 1. OCR Provider Configuration

- `OCR_PROVIDER` env var now supports: `mock` (default), `deepseek`, `openai`, `tesseract`
- `OCR_MODEL` env var: provider-independent model override. Falls back to `DEEPSEEK_OCR_MODEL` or `OPENAI_OCR_MODEL` for backward compatibility
- Missing API key returns clear Turkish error instead of silent fallback
- No hardcoded model names — all model values are configurable
- `TesseractOcrProvider`: local-only OCR using Tesseract.js, no API key needed. Lower accuracy; recommended as fallback

### 2. OCR Upload Hardening

- Max image size: 8 MB (enforced client-side and server-side)
- HEIC-to-JPEG conversion size check after conversion
- Supported MIME types: `image/jpeg`, `image/png`, `image/webp`, `image/heic`, `image/heif`
- **multipart/form-data** upload support alongside existing JSON body (data URL)
- Content-length pre-check (12 MB max request body)
- Friendly Turkish error messages for all validation failures

### 3. Vehicle Duplicate Protection

- `Vehicle(workshopId, plate)` unique constraint at DB level (existing, preserved)
- `Vehicle(workshopId, vin)` indexed for fast lookups (existing, preserved)
- VIN duplicate check in confirm route: warns if VIN matches another vehicle in workshop
- Confirm route returns `warnings` array with plate/VIN conflict messages
- Vehicle create action catches unique constraint errors with Turkish message
- Documented cleanup path for existing duplicates in migration notes

### 4. Smart Capture UX

- Per-field confidence percentage shown next to labels
- Low-confidence fields (< 70%) highlighted with amber border and warning text
- OCR provider label shown in confirm step (e.g., "Demo (Mock)", "OpenAI Vision", "Tesseract + DeepSeek", "Tesseract (Yerel)")
- `rawText` excluded from API responses — stored in DB only, never sent to client
- Provider name included in API response (`provider` field)
- User confirmation remains mandatory — never auto-saves unconfirmed data

### 5. OCR Data Safety

- `OcrLog.rawText` field documented as sensitive (never exposed via public API)
- `OcrLog.extractedJson` and `confirmedJson` are workshop-scoped (queried by `workshopId`)
- No internal IDs exposed in client-facing API responses
- Audit log entries created for `ocr_capture` and `ocr_confirmed` actions

### 6. Tesseract Lifecycle Cleanup

- Singleton worker pattern with queue-based sequential recognition
- `terminateTesseractWorker()` exported for graceful shutdown
- Lifecycle limitation documented: worker may not persist across serverless invocations
- Error handling: worker promise cleared on initialization failure for retry

### 7. Integration Points

- `/app/smart-capture/registration` — full flow with confidence display and provider label
- `/app/orders/new` — "Ruhsattan Doldur" card (existing, unchanged)
- `/app/vehicles/new` — "Ruhsattan Oku" link (existing, unchanged)
- `/app/intakes/new?source=registration` — prefill flow (existing, unchanged)
- Duplicate plate/VIN warning flow in confirm route

## Schema Changes

No new schema changes in v0.3.3. The following existing constraints are preserved:

- `Vehicle(workshopId, plate)` — unique constraint
- `Vehicle(workshopId, vin)` — index (not unique, VIN can be null)
- `OcrLog` — workshop-scoped audit model with `rawText`, `extractedJson`, `confirmedJson`

## Known Limitations

1. `OCR_PROVIDER` defaults to `mock` — returns fixed demo data. Real OCR requires `deepseek`, `openai`, or `tesseract` with API keys (where applicable)
2. Tesseract-only provider (`tesseract`) has lower accuracy than DeepSeek/OpenAI; best used as fallback or testing
3. Both `DEEPSEEK_OCR_MODEL`/`OPENAI_OCR_MODEL` are required when using their respective providers if `OCR_MODEL` is not set (no hardcoded defaults)
4. Tesseract.js worker is a lazy singleton; `terminateTesseractWorker()` is exported for graceful shutdown but not called automatically
5. Large images sent as base64 data URLs in JSON body (multipart alternative now available); server-side content-length check caps at 12 MB
6. `vinOcrRaw` field on Vehicle model is defined but not populated in v0.3.3
7. OpenAI provider uses Responses API (`/v1/responses`) — requires compatible model
8. HEIC conversion adds processing overhead; preview data URL stored in memory
9. VIN duplicate check is application-level (not DB unique constraint) because VIN can be null/empty
10. Currency is TRY only
11. No real SMS/WhatsApp/payment/invoice integration (all mocks or client-side links)
