# BakimX v0.1.4 — İş Emri UX Alignment

**Release Date:** June 2026
**Previous Release:** v0.1.3 — Image Storage & Intake Media Foundation

---

## Summary

v0.1.4 redesigns the BakimX work order experience end-to-end so the product feels like a professional auto service operations tool. The protected app shell gets a dark navy sidebar with grouped navigation, a topbar with global search and quick “Yeni İş Emri” CTA, and a mobile bottom navigation. The Work Orders list is rebuilt with KPI cards, filters, a desktop table and a mobile card layout. The Work Order detail page is reorganized into a header, status controls, customer/vehicle summary, parts/labor, complaint/notes, damage/photos, a sticky pricing panel, and share/export actions. New schema fields (workOrderNo, paymentStatus, technician, estimated delivery, discount, tax) and shared `formatOrderSummary`/`calculateOrderTotals` helpers are added with full backward compatibility. New placeholder routes (Teklifler, Randevular, Stok, Tedarikçiler, Kasa, Raporlar) are stubbed as “Yakında” — no Docker, no real OCR, no real SMS, no real payment module.

## Changes

### 1. App Shell — Dark Navy Sidebar & Grouped Navigation (`components/app/app-shell.tsx`)

- Dark navy sidebar (`bg-[#0F172A]`) on desktop, slide-in drawer on mobile
- Workshop name (or "BakimX") at the top with brand mark
- **Grouped navigation modules:**
  - **Ana Panel:** Genel Bakış
  - **Servis:** İş Emirleri, Araç Kabulleri, Müşteriler, Araçlar, Teklifler (Yakında), Randevular (Yakında), Bakım Hatırlatmaları (Yakında)
  - **Depo & Finans:** Stok / Parçalar (Yakında), Tedarikçiler (Yakında), Kasa (Yakında)
  - **Analiz:** Raporlar (Yakında)
  - **Ayarlar:** İş Yeri Profili
- Topbar:
  - Page title (large enough on desktop, responsive)
  - Global search input with placeholder “Plaka, müşteri, iş emri ara” — routes to `/app/orders?q=...`
  - Quick CTA: “+ Yeni İş Emri” (visible on desktop, icon-only on mobile)
  - Notification icon (placeholder)
  - User menu (logout)
- Mobile:
  - Sidebar becomes a slide-in drawer
  - Bottom navigation with 4 main modules (Panel, İş Emirleri, Kabuller, Müşteriler)
  - “+ Yeni İş Emri” floating icon in topbar
- Active link highlighting with subtle blue accent
- Sidebar links auto-close mobile drawer on click

### 2. Work Orders List Page (`app/app/orders/page.tsx`, `components/app/order-list.tsx`)

- **Page title** “İş Emirleri” with **breadcrumb** “Ana Panel / İş Emirleri”
- **Primary CTA** “+ Yeni İş Emri” (desktop), floating icon (mobile)
- **KPI cards** with counts and clickable filter behavior:
  - Aktif, Tamamlandı (Teslime Hazır), Teslim Edildi, İptal
- **Filters** via GET form:
  - Text search (İş emri no, plaka, müşteri adı/telefon)
  - Status filter (all 8 statuses)
  - Payment status filter (unpaid / partial / paid)
- **Desktop table** (lg+): İş No, Plaka/Araç, Müşteri, Teknisyen, Durum, Ödeme, Toplam, Giriş, Tahmini Teslim, İşlem
- **Mobile cards**: WorkOrderNo, PlateBadge, customer, status/payment badges, total, estimated delivery
- **PlateBadge** component: dark navy monospace badge for plates (consistent style)
- **StatusBadge** & **PaymentBadge** components with Turkish labels and color tokens
- Tenant-isolated: every query is scoped by `workshopId`

### 3. New Work Order Page (`app/app/orders/new/page.tsx`, `components/app/new-order-selector.tsx`)

- Replaces “Henüz iş emri yok” dead-end with a real entry point
- Two clear paths:
  - **Yeni Araç Kabulü** (links to existing intake wizard) for completely new customers
  - **Kabullerden İş Emri Oluştur** — list of recent intakes that are eligible (approved / in_progress / ready_for_delivery / delivered) without an existing order
  - Each row shows: PlateBadge, customer, vehicle, status badge, work-order availability
  - Clicking a row creates a ServiceOrder via `/api/orders` and redirects to detail
  - Other (ineligible) intakes are listed below for context (go to existing order, or first approve)

### 4. Work Order Detail Page (`app/app/orders/[id]/page.tsx`, `components/app/order-detail.tsx`)

Major rewrite. Two-column on desktop (lg+), single column on mobile. Sticky pricing summary on desktop.

Sections:
- **Header**: PlateBadge, StatusBadge, PaymentBadge, customer name, vehicle summary
- **Quick actions**: Contextual “Durumu Güncelle” buttons driven by allowed transition map (no obviously invalid UI states)
- **Müşteri & Araç** card: customer (phone/email), vehicle (brand/model/year/mileage/VIN), links to intake detail
- **Kullanılan Parçalar & İşçilikler** card: grouped list, inline add form, remove, support for SKU, birim, not
- **Şikayet & Notlar** card: customer complaint (visible on public output) and technician internal note (clearly marked as “Müşteri çıktısında gösterilmez”)
- **Araç Hasar Haritası** summary: reuses intake damage marks; full editor accessible via link to intake detail
- **Fotoğraflar** summary: 8-thumbnail grid; full gallery via intake detail
- **Fiyatlandırma** (sticky on desktop): parts/labor totals, subtotal, discount, KDV, **Genel Toplam** (grand total); inline editing of discount and KDV rate
- **İş Emri Bilgileri**: workOrderNo, creation date, estimated delivery, technician, notes, payment status
- **Müşteri Çıktısı**: public link, PDF/print, WhatsApp share (uses existing helper)

Allowed status transitions:
- `draft` → `in_progress` | `waiting_approval` | `cancelled`
- `waiting_approval` → `approved` | `in_progress` | `cancelled`
- `approved` → `in_progress` | `waiting_parts`
- `in_progress` → `waiting_parts` | `ready_for_delivery`
- `waiting_parts` → `in_progress` | `ready_for_delivery`
- `ready_for_delivery` → `delivered` | `cancelled`
- `delivered` and `cancelled` are terminal (no further buttons)

### 5. Status & Payment UX

- New `OrderStatus` enum values: `waiting_approval`, `approved`, `waiting_parts` (added in schema)
- New `PaymentStatus` enum: `unpaid` (default), `partial`, `paid`, `cancelled`
- New API routes:
  - `POST /api/orders/[id]/payment` — update payment status
  - `POST /api/orders/[id]/meta` — update technician, estimated delivery, discount, tax, notes
- New server actions in `app/app/orders/actions.ts`:
  - `updateOrderPaymentStatusAction`
  - `updateOrderMetaAction`
  - `addOrderItemAction` extended to accept `sku` and `unit`
- Status changes are audit-logged (existing pattern, unchanged)

### 6. Work Order Number Convention

- New helper `lib/work-order-number.ts`:
  - `formatWorkOrderNo(order)` — uses `workOrderNo` if set, otherwise falls back to `BX-{last6charsOfCuid}` so old records still display a stable readable number
  - `generateWorkOrderNo()` — generates a new number (e.g. `BXK7Y2P9`) on creation
- `createServiceOrderAction` now assigns a `workOrderNo` automatically
- Backward compatible: existing orders with `workOrderNo = null` still render correctly

### 7. Pricing / Totals

- `lib/totals.ts` extended:
  - `calculateOrderTotals(items, { discountAmount, taxRate })` — single source of truth for parts/labor/subtotal/discount/tax/grand total
  - `formatOrderSummary(items, options)` — formatted strings with `—` for empty values
- Pricing panel uses these helpers; KDV rate is shown as percent (derived)
- “İskonto & KDV” inline editor; live preview total
- No invoice/e-archive/e-invoice — purely display totals

### 8. ServiceOrderItem Schema Extension

- New fields: `sku: String?`, `unit: String?`
- Backward compatible (nullable)
- Add-item form supports these fields

### 9. Global Search Preparation

- Topbar search is functional
- Submits to `/app/orders?q=...` and matches against workOrderNo, plate, customer name/phone
- Full cross-module global search is out of scope (kept lightweight)

### 10. Placeholder Routes (“Yakında”)

- New shell component `components/app/coming-soon-shell.tsx` with workshop context
- Placeholder pages for: `/app/quotes`, `/app/appointments`, `/app/reminders`, `/app/inventory`, `/app/suppliers`, `/app/cash`, `/app/reports`
- Each shows a friendly “Yakında” notice; not implemented, no fake data, no broken links
- Sidebar items clearly show a “Yakında” badge

### 11. Tenant Isolation & Security (unchanged contract)

- Every new query/mutation is scoped by `workshopId`
- New payment/meta actions follow the same `requireAuth` + scoped Prisma pattern
- No new exposure of internal data

### 12. Regression-Safe

- All v0.1.3 routes still render
- Landing, login, register, customers, vehicles, intakes, intake detail, public output, photos still work
- Mock SMS approval flow unchanged
- Storage provider abstraction (mock / Supabase / S3 placeholder) unchanged
- No new env vars required
- No Docker files added

- Plate scan button is a placeholder only (no real OCR integration)
- Voice fill button is a placeholder only (no real speech-to-text)
- Barcode scan button is a placeholder only (no real barcode integration)
- No real SMS provider (mock/demo only)
- No real payment module (payment status is a label only)
- No invoice/e-archive/e-invoice
- No multi-branch enterprise module
- No Docker
- S3 storage provider still placeholder
- HEIC files still rejected
- @react-pdf/renderer not yet generating server-side PDF
