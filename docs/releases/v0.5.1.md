# BakimX v0.5.1 — Calendar & Reminder Automation

**Release Date:** June 2026

## Overview

This release adds calendar provider abstraction, Google Calendar foundation, reminder automation scheduling, and a unified calendar view for appointments, deliveries, and maintenance reminders.

## New Features

### 1. Calendar Provider Abstraction (`src/lib/calendar/`)

Abstracted provider system supporting hot-swappable calendar providers via environment variables:

- **Mock:** `MockCalendarProvider` (default) — in-memory calendar, console logging
- **Google:** `GoogleCalendarProvider` — Google Calendar API v3 integration (foundation)

Environment variables:
- `CALENDAR_PROVIDER=mock|google` (default: mock)
- `GOOGLE_CALENDAR_ID` (required when provider=google)
- `GOOGLE_CALENDAR_ACCESS_TOKEN` (required when provider=google)
- `GOOGLE_CALENDAR_API_URL` (optional, defaults to `https://www.googleapis.com/calendar/v3`)

### 2. Calendar View (`/app/calendar`)

Unified calendar view showing three event types:
- **Randevular (Appointments)** — Blue indicators
- **Teslimatlar (Deliveries)** — Green/emerald indicators
- **Bakım Hatırlatmaları (Maintenance Reminders)** — Amber indicators

View modes:
- Günlük (Day) — Single day timeline
- Haftalık (Week) — 7-day grid with hourly events
- Aylık (Month) — Full month grid with compact event indicators

Navigation: Previous/Next period, Today button

### 3. Google Calendar Foundation

- Provider abstraction follows existing pattern (SMS/WhatsApp/Email/Storage/OCR/AI)
- Google Calendar API v3 REST integration
- No crash without credentials — falls back to mock if keys missing
- Create, update, delete, and list events supported

### 4. Calendar Settings (`/app/settings/calendar`)

- Shows active calendar provider
- Google Calendar credential status
- Configuration instructions with .env examples
- Sync behavior documentation

### 5. Reminder Automation

**"Hatırlatmaları Kontrol Et" button** on calendar page:
- Manually triggers all reminder jobs
- Runs appointment reminder processing (24h/1h before)
- Runs maintenance reminder processing (due/upcoming/overdue)
- Runs delivery reminder processing (orders ready for delivery with unpaid balance)
- Results displayed inline on calendar page

**Job types:**
- `appointment_reminder` — Appointment notifications via existing communication providers
- `maintenance_reminder` — Maintenance reminder notifications
- `delivery_reminder` — Delivery + payment reminder notifications for ready orders

### 6. Scheduled Job Foundation (`src/lib/calendar/reminder-scheduler.ts`)

- Safe scheduler abstraction — no external queue required
- Each job execution logged to `ReminderExecutionLog` table
- Deduplication via `lastRemindedAt` (inherited from v0.5.0)
- Manual trigger via calendar page button
- Cron endpoint `/api/cron/reminders` still available for external schedulers

### 7. Calendar Sync Logging

All calendar sync operations logged to `CalendarSyncLog` table:
- Provider name
- Direction (outbound/inbound)
- Status (success/failed/pending)
- Entity type and ID
- External event ID (from Google Calendar)
- Error messages

### 8. Reminder Execution Logging

All reminder job runs logged to `ReminderExecutionLog` table:
- Job type (appointment_reminder, maintenance_reminder, delivery_reminder)
- Status (success/partial/failed)
- Processed, sent, failed counts
- Error messages

### 9. Calendar Sync Integration

Calendar sync triggered automatically on:
- Appointment creation → `syncAppointmentToCalendar()`
- Order delivery date set → `syncDeliveryToCalendar()`
- Maintenance reminder creation (with due date) → `syncMaintenanceReminderToCalendar()`

### 10. Security

- All calendar operations are workshop-scoped
- No secrets exposed in sync logs or UI
- Google Calendar credentials only in environment variables
- Customer communication preferences respected (inherited from v0.5.0)
- Fallback to mock provider if Google credentials missing — no crash

## Database Changes

New models:
- `CalendarSyncLog` — Calendar synchronization audit log (provider, direction, status, external event ID)
- `ReminderExecutionLog` — Reminder job execution log (job type, processed/sent/failed counts)

Run `bunx prisma db push` to apply schema changes.

## Breaking Changes

None. All new features are additive. Calendar provider defaults to mock with no external configuration required.

## Limitations

- Google Calendar provider requires manual OAuth token — no auto-refresh flow yet
- Calendar sync is one-way (outbound only) — no import from Google Calendar
- Reminder scheduling is manual (button) or via external cron — no internal scheduler
- Calendar events from Google Calendar list API are not yet merged into the calendar view
- Rate limiting for calendar API calls not yet implemented
